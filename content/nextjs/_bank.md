## Interview Questions

### ¿Por qué el App Router usa Server Components por defecto y qué implica para el bundle del cliente?
Porque el modelo de componentes del servidor permite mover la lógica de data fetching, acceso a DB y transformación de datos al servidor, sin que ninguno de ese código llegue al bundle del browser. El resultado es un bundle del cliente significativamente más pequeño: las dependencias usadas solo en Server Components (ORMs, librerías de parsing, SDKs de terceros) nunca se incluyen. Esto mejora directamente el Time to Interactive y reduce el costo de parsing de JS en dispositivos lentos. `nextjs.org/docs/app/building-your-application/rendering/server-components`

### ¿Cuál es la diferencia entre `layout.tsx` y `template.tsx`?
`layout.tsx` persiste entre navegaciones del mismo segmento: el componente se monta una sola vez y no se re-renderiza cuando el usuario navega entre hijos del mismo layout. `template.tsx` crea una nueva instancia en cada navegación, re-ejecutando efectos y reiniciando el estado. Usás `template.tsx` cuando necesitás animaciones de entrada/salida por página, o cuando el estado del layout NO debe persistir entre rutas hijas. El 95% de los casos es `layout.tsx`. `nextjs.org/docs/app/building-your-application/routing/pages-and-layouts`

### ¿Qué es un route group y cuándo lo usás?
Un route group es una carpeta entre paréntesis `(nombre)` que agrupa rutas bajo el mismo directorio sin añadir ese segmento a la URL. Se usan para aplicar distintos `layout.tsx` a grupos de rutas que comparten el mismo path prefix (ej: `(auth)/login` y `(auth)/register` comparten un layout de auth, pero `(marketing)/about` tiene un layout diferente, sin que `(auth)` o `(marketing)` aparezcan en la URL). También sirven para organizar código sin afectar el routing. `nextjs.org/docs/app/building-your-application/routing/route-groups`

### ¿Qué ocurre con el caché cuando hacés un `redirect()` dentro de una Server Action?
`redirect()` lanza un error especial de Next.js que interrumpe la ejecución de la función. Si necesitás invalidar caché antes de redirigir, debés llamar `revalidatePath()` o `revalidateTag()` antes del `redirect()`, porque ningún código después del `redirect()` se ejecuta. Este es un gotcha común: si ponés el `revalidatePath` después del `redirect`, la caché nunca se invalida. `nextjs.org/docs/app/building-your-application/data-fetching/server-actions-and-mutations`

### ¿Por qué no podés usar `async/await` directamente en un Client Component para fetchear datos?
Los Client Components se ejecutan en el browser donde `async` en el cuerpo del componente causaría un race condition con el ciclo de vida de React. La solución nativa es `useEffect` con estado local o librerías de data fetching como TanStack Query/SWR que manejan el ciclo de vida, loading y error. La alternativa correcta en App Router es mover el fetching a un Server Component padre y pasar la data como props, o usar un React Server Component asíncrono para el fetching y suspender el Client Component hijo. `nextjs.org/docs/app/building-your-application/data-fetching/fetching`

### ¿Qué es Request Memoization y cómo afecta al diseño de componentes?
Request Memoization es una deduplicación automática de llamadas `fetch` idénticas (misma URL + opciones) dentro del mismo render tree de un request. Si tres componentes distintos en el mismo render llaman `fetch('/api/user')`, Next.js solo ejecuta la request una vez y reutiliza el resultado. Esto habilita un patrón de diseño importante: podés hacer fetch del mismo dato en múltiples componentes sin preocuparte por waterfalls ni duplicación de requests, cada componente declara su propia dependencia de data. `nextjs.org/docs/app/building-your-application/caching#request-memoization`

### ¿Cuándo usás `loading.tsx` vs `<Suspense>` manual?
`loading.tsx` agrega automáticamente un Suspense boundary que envuelve toda la página (`page.tsx`) del segmento. Es conveniente para mostrar un skeleton mientras toda la ruta carga. `<Suspense>` manual te da granularidad: podés tener partes de la página disponibles inmediatamente mientras otras cargan con su propio fallback. Si tu página tiene una sección de datos rápidos y otra lenta (ej: main content + recommendations), `<Suspense>` manual en la sección lenta es la opción correcta, el usuario ve contenido útil antes. `nextjs.org/docs/app/building-your-application/routing/loading-ui-and-streaming`

### ¿Qué hace `generateStaticParams` y por qué reemplaza a `getStaticPaths`?
`generateStaticParams` le dice a Next.js qué parámetros dinámicos pre-generar en build time como páginas estáticas. Retorna un array de objetos con los params de la ruta (ej: `[{ slug: 'post-1' }, { slug: 'post-2' }]`). Reemplaza a `getStaticPaths` del Pages Router con una API más simple: no necesita `fallback` como prop separada (se controla con `export const dynamicParams = false/true`). Si `dynamicParams` es `true` (default), los params no generados en build se generan on-demand en el primer request. `nextjs.org/docs/app/api-reference/functions/generate-static-params`

### ¿Qué es `unstable_cache` y cuándo lo necesitás?
`unstable_cache` envuelve cualquier función async con las mismas semánticas de caché que `fetch` (revalidación por tiempo o por tag). Lo necesitás cuando hacés queries a la base de datos directamente (sin `fetch`) y querés que el resultado sea cacheado y revalidable. Sin `unstable_cache`, una query de Prisma en un Server Component se ejecuta en cada request si la ruta es dinámica. Con `unstable_cache`, el resultado se guarda en el Data Cache y se puede invalidar con `revalidateTag`. `nextjs.org/docs/app/api-reference/functions/unstable_cache`

### ¿Cuál es la diferencia entre `error.tsx` y `global-error.tsx`?
`error.tsx` captura errores dentro de su segmento de ruta y sus hijos, pero NO errores en el `layout.tsx` del mismo nivel (porque el Error Boundary es hijo del layout). `global-error.tsx` en la raíz de `app/` captura errores del root layout y del root template, es el último recurso para errores críticos que rompen el shell de la aplicación. `global-error.tsx` debe incluir sus propias etiquetas `<html>` y `<body>` porque reemplaza al root layout cuando se activa. `nextjs.org/docs/app/building-your-application/routing/error-handling`

### ¿Cuándo tiene sentido usar Parallel Routes?
Las Parallel Routes (`@slot`) permiten renderizar múltiples segmentos de página independientes dentro del mismo layout, cada uno con su propio loading, error y sub-navegación. El caso de uso más claro es un dashboard con paneles que cargan de forma independiente (tabla + gráfico + stats), o un inbox con panel de lista + panel de detalle que pueden mostrar estados distintos simultáneamente. A diferencia de mostrar múltiples componentes en un layout normal, cada slot tiene su propio árbol de routing con `loading.tsx` y `error.tsx` independientes. `nextjs.org/docs/app/building-your-application/routing/parallel-routes`

### ¿Qué son las Intercepting Routes y cuál es su caso de uso principal?
Las Intercepting Routes permiten capturar una URL que normalmente renderizaría una página completa y mostrarla en el contexto de la ruta actual (ej: en un modal). Se definen con la convención `(.)` (mismo nivel), `(..)` (un nivel arriba), etc. El caso clásico es un feed de fotos: al hacer clic en una foto, la URL cambia a `/photos/123` y se muestra en un modal; si el usuario refresca o llega directamente a esa URL, ve la página completa de la foto. Permite URLs compartibles sin perder el contexto de navegación. `nextjs.org/docs/app/building-your-application/routing/intercepting-routes`

### ¿Por qué el middleware corre en Edge runtime y qué significa eso en práctica?
El Edge runtime está distribuido globalmente en nodos de red cercanos al usuario (similar a un CDN), lo que da latencias muy bajas. Pero no tiene el runtime completo de Node.js: no hay `fs`, no hay TCP connections directas, no podés usar Prisma sin edge adapter, y las APIs de crypto vienen de la Web Crypto API, no de Node. En práctica: en middleware solo podés verificar JWTs con Web Crypto, leer/escribir cookies, y hacer fetch a APIs HTTP. Para cualquier otra lógica que requiera Node.js, necesitás un Route Handler en Node runtime. `nextjs.org/docs/app/building-your-application/routing/middleware`

### ¿Cuál es la diferencia entre `cookies()` en un Server Component vs en un Route Handler?
En ambos casos usás `cookies()` de `next/headers`, pero el contexto cambia cómo se aplican las mutaciones. En Server Components, `cookies()` es de solo lectura, no podés `set` porque el Header ya fue enviado. Para escribir cookies, necesitás una Server Action o un Route Handler donde podés hacer `cookies().set(...)` porque el response no fue comprometido. Este es un gotcha frecuente: intentar `cookies().set()` en un Server Component lanza un error en runtime. `nextjs.org/docs/app/api-reference/functions/cookies`

### ¿Qué hace el `matcher` en middleware y cómo lo optimizás?
El `matcher` es un array de patrones que define sobre qué rutas se ejecuta el middleware. Sin `matcher`, el middleware corre en TODAS las requests incluyendo assets estáticos (`_next/static`, imágenes, fonts), lo que añade latencia innecesaria. La configuración correcta excluye assets estáticos con una expresión negativa. El matcher usa la sintaxis de `path-to-regexp`. También podés usar `matcher` para incluir solo las rutas protegidas (ej: `/dashboard/:path*`) en lugar de excluir las públicas, elige el enfoque según cuál lista es más corta. `nextjs.org/docs/app/building-your-application/routing/middleware#matcher`

### ¿Cuál es la estrategia de sesión recomendada en Auth.js con App Router y por qué?
JWT (strategy: "jwt") es el default y el más común con App Router porque no requiere una query a la base de datos para cada request, el token se verifica criptográficamente. Esto es crítico en middleware, donde no podés tener conexiones de DB. La desventaja: no podés invalidar un JWT antes de que expire sin una blacklist (que requiere storage). DB sessions (strategy: "database") son invalidables instantáneamente (logout real) pero requieren una query por request. Para la mayoría de las apps, JWT con un TTL corto (15-60 min) + refresh token rotativo es el balance adecuado. `nextjs.org/docs/app/building-your-application/authentication`

### ¿Cuándo deberías usar `headers()` en un Server Component y qué side effect tiene?
`headers()` de `next/headers` permite leer los headers del request HTTP (ej: User-Agent, Accept-Language, Authorization). Al usarlo en un Server Component, la ruta se vuelve automáticamente dinámica (Next.js la marca como "dynamic function call"), desactivando el Full Route Cache. Tiene sentido cuando necesitás personalizar la respuesta basándote en headers del request. Si solo lo necesitás en un subcomponente, podés leer el header en el Server Component padre y pasarlo como prop, para que el resto del árbol pueda seguir siendo estático. `nextjs.org/docs/app/api-reference/functions/headers`

### ¿Cómo implementarías protección de rutas sin usar Auth.js?
Verificando manualmente el token de sesión con `cookies()` en middleware o en el Server Component. En middleware: leer la cookie de sesión, verificar el JWT con `jose` (compatible con Web Crypto API / Edge runtime), y redirigir si no es válido. En Server Component: usar `cookies()` para leer el token, verificarlo, y llamar `redirect('/login')` si falla. La ventaja del middleware es la centralización (no repetís la verificación en cada página); la desventaja es que solo podés usar APIs edge-compatible. `nextjs.org/docs/app/building-your-application/authentication#session-management`

### ¿Qué es Partial Prerendering (PPR) y en qué se diferencia de SSR + Suspense?
PPR pre-renderiza el shell estático de la página en build time y hace streaming de las partes dinámicas en runtime, todo en un único request HTTP. La diferencia con SSR + Suspense convencional es que en SSR puro el servidor procesa todo el request en runtime (más lento); con PPR, el shell estático se sirve instantáneamente desde el Full Route Cache y solo las "holes" dinámicas (Suspense boundaries con content dinámico) se resuelven en runtime. El usuario ve el shell de la página inmediatamente y el contenido dinámico aparece en streaming. Es experimental en Next.js 15. `nextjs.org/docs/app/building-your-application/rendering/partial-prerendering`

### ¿Por qué `fetch` en un Server Component no necesita `useEffect`?
Porque los Server Components son funciones async que se ejecutan una sola vez en el servidor, no son parte del ciclo de vida de React en el cliente. Podés usar `await` directamente en el cuerpo del componente porque no hay reconciler, no hay renders subsiguientes, no hay riesgo de memory leaks. `useEffect` existe para sincronizar con el DOM o APIs del browser después del render, nada de eso aplica en el servidor. El modelo mental es: un Server Component es como una función async de Node.js que retorna HTML, no un componente React tradicional. `nextjs.org/docs/app/building-your-application/data-fetching/fetching`

### ¿Cuándo usarías ISR en lugar de SSG puro?
ISR cuando el contenido cambia con cierta frecuencia pero no en tiempo real: posts de blog que se actualizan cada hora, páginas de producto con stock que cambia, precios. SSG puro para contenido verdaderamente estático (docs, landing pages, páginas legales). La ventaja de ISR es que las builds no necesitan regenerar todo el sitio cuando cambia un dato, solo el segmento afectado se regenera en background al expirar el TTL. Si necesitás invalidación inmediata ante cambios específicos, ISR con `revalidateTag` (on-demand revalidation) es más preciso que el TTL basado en tiempo. `nextjs.org/docs/app/building-your-application/data-fetching/incremental-static-regeneration`

### ¿Cómo implementarías un webhook de Stripe en Next.js App Router?
Con un Route Handler en `app/api/webhooks/stripe/route.ts` que exporte `POST`. El body debe leerse como `text()` (no como JSON) para que la verificación de firma de Stripe funcione correctamente sobre el payload raw. La firma se verifica con `stripe.webhooks.constructEvent(body, sig, secret)`. La ruta debe excluirse del middleware de auth (si lo hay) ya que Stripe no envía cookies de sesión. También hay que deshabilitar cualquier body parser adicional, los Route Handlers no tienen body parser por defecto, lo que es correcto. `nextjs.org/docs/app/building-your-application/routing/route-handlers`

### ¿Qué es `dynamicParams` y cómo se relaciona con `generateStaticParams`?
`dynamicParams` es una variable de segmento que controla qué pasa cuando alguien accede a un param que NO fue generado por `generateStaticParams`. Con `export const dynamicParams = true` (default), esos params se generan on-demand al primer request y se cachean. Con `dynamicParams = false`, Next.js retorna 404 para cualquier param no pre-generado. Usás `false` cuando tenés un conjunto fijo y conocido de rutas (ej: páginas de países ISO) y querés evitar que rutas arbitrarias sean servidas. `nextjs.org/docs/app/api-reference/file-conventions/route-segment-config#dynamicparams`

### ¿Cuál es la diferencia entre el Full Route Cache y el Router Cache?
El **Full Route Cache** vive en el servidor: guarda el HTML pre-renderizado + RSC payload de segmentos estáticos, persiste entre deployments (en Vercel) y se invalida con `revalidatePath` o al hacer un nuevo deploy. El **Router Cache** vive en el browser del usuario: guarda los RSC payloads de rutas visitadas para navegación instantánea back/forward, dura 30 segundos para páginas dinámicas y 5 minutos para estáticas, y se limpia al navegar fuera o al expirar. Son capas independientes con distintos mecanismos de invalidación. `nextjs.org/docs/app/building-your-application/caching#router-cache`

### ¿Cómo evitarías exponer un secret de API en el cliente usando el App Router?
Manteniendo el fetching exclusivamente en Server Components o Server Actions. Un Server Component puede importar variables de entorno sin el prefijo `NEXT_PUBLIC_` y usarlas directamente, esas variables nunca llegan al bundle del cliente. Si un Client Component necesita data de una API que requiere un secret, la solución es crear un Server Component que fetchee la data y la pase como props al Client Component, o usar una Server Action como intermediario. Nunca expongas secrets en `NEXT_PUBLIC_` ni en código client-side. `nextjs.org/docs/app/building-your-application/rendering/server-components`

### ¿Qué pasa con el estado local de un layout cuando navegás entre sus hijos?
El estado del layout **persiste**: el componente no se desmonta ni remonta al navegar entre rutas hijas. Esto es una de las ventajas clave del App Router frente al Pages Router donde `_app.js` también persistía pero con menos control. Si tenés un `useState` en un layout para un sidebar abierto/cerrado, ese estado sobrevive la navegación entre las rutas hijas. La excepción es `template.tsx`, que crea una nueva instancia en cada navegación, reiniciando el estado. `nextjs.org/docs/app/building-your-application/routing/pages-and-layouts#layouts`

### ¿Cuándo tiene sentido agregar `"use client"` a nivel de directorio vs por archivo?
Nunca a nivel de directorio, la directiva `"use client"` solo funciona por archivo. Si varios archivos del mismo directorio necesitan ser Client Components, cada uno debe tener la directiva. Sin embargo, el patrón recomendado es minimizar el número de archivos con `"use client"` creando pocos Client Components "raíz" que encapsulan la interactividad y pasando Server Components como `children`. La alternativa de marcar todo como cliente por conveniencia destruye el beneficio del App Router. `nextjs.org/docs/app/building-your-application/rendering/client-components`

### ¿Cómo implementarías i18n en Next.js App Router sin librería externa?
Usando middleware para detectar el locale del usuario (desde `Accept-Language` header o cookie) y redirigir/reescribir a `/[locale]/...`. Estructurando las rutas bajo `app/[locale]/` con un layout que cargue los diccionarios de traducciones del servidor. Los diccionarios se importan dinámicamente en el layout según el param `locale`, sin mandarlos todos al cliente. Esta es la aproximación que muestra la documentación oficial de Next.js para i18n en App Router. Para proyectos grandes, librerías como `next-intl` encapsulan este patrón. `nextjs.org/docs/app/building-your-application/routing/internationalization`

### ¿Qué es `next/dynamic` y en qué se diferencia de `import()` estándar?
`next/dynamic` es un wrapper sobre `React.lazy` + `Suspense` con funcionalidades adicionales específicas de Next.js: la opción `ssr: false` para evitar que el componente se renderice en el servidor (útil para componentes que usan APIs del browser como `window`), la prop `loading` para el fallback, y mejor integración con el sistema de chunks de Next.js. `import()` estándar con `React.lazy` también funciona, pero sin `ssr: false` el componente intentará renderizarse en el servidor y fallará si usa APIs del browser. `nextjs.org/docs/app/building-your-application/optimizing/lazy-loading`

### ¿Cuál es el impacto de usar `next/font` en Core Web Vitals?
`next/font` elimina los requests externos a Google Fonts (que bloquean el render), hace subsetting automático reduciendo el tamaño del archivo, y agrega `font-display: optional` o `swap` para controlar el FOUT. El impacto en CLS es el más significativo: al self-hostear la fuente y reservar el espacio en build time con CSS variables, no hay layout shift cuando la fuente carga. El impacto en LCP es indirecto: menos requests de terceros bloqueantes = el LCP content carga antes. También mejora el score de performance de Lighthouse al eliminar "Eliminate render-blocking resources" de Google Fonts. `nextjs.org/docs/app/building-your-application/optimizing/fonts`

### ¿Cuándo deployarías Next.js en self-hosted vs Vercel?
Vercel cuando querés zero-config: ISR, Edge Middleware, optimización de imágenes, analytics, preview deployments y rollbacks funcionan out-of-the-box. Self-hosted (Docker + Node.js con `output: 'standalone'`) cuando tenés restricciones de compliance/datos (no podés usar cloud externo), costos a escala muy alta, o necesitás infraestructura existente (Kubernetes, AWS ECS). La limitación más importante del self-hosting es que ISR y el Full Route Cache funcionan por instancia, en un cluster multi-instancia, el caché no se comparte y necesitás un adapter (ej: el adapter de Vercel para Redis). `nextjs.org/docs/app/building-your-application/deploying`

### ¿Cómo funciona el streaming en App Router y por qué mejora la percepción de performance?
El streaming envía el HTML al browser progresivamente: primero el shell de la página (lo que no necesita data), luego las partes dentro de `<Suspense>` a medida que sus promesas resuelven. El browser puede pintar y hacer interactivo el contenido disponible sin esperar a que toda la data esté lista. Esto mejora el First Contentful Paint (FCP) y la percepción de velocidad aunque el Time to Last Byte sea el mismo. `loading.tsx` es azúcar sintáctico que activa este mecanismo automáticamente para toda la página, mientras que `<Suspense>` manual lo activa para secciones específicas. `nextjs.org/docs/app/building-your-application/routing/loading-ui-and-streaming`

### ¿Qué es `useFormStatus` y por qué necesitás un componente separado para usarlo?
`useFormStatus` lee el estado del `<form>` padre más cercano del DOM, específicamente si hay un submission pendiente. Debe usarse en un componente que sea hijo del form, NO en el mismo componente que define el form, porque necesita estar dentro del árbol del form para acceder a su contexto. Por eso el patrón estándar es extraer el botón de submit a su propio componente (`<SubmitButton>`) que internamente usa `useFormStatus` para deshabilitarse durante el pending. `nextjs.org/docs/app/building-your-application/data-fetching/server-actions-and-mutations`

### ¿Cuál es la diferencia entre `useActionState` y `useState` para manejar el resultado de una Server Action?
`useActionState` está diseñado para integrar Server Actions con formularios: acumula el estado de la última llamada (resultado o error), maneja automáticamente el pending, y funciona sin JavaScript habilitado en el cliente (Progressive Enhancement). `useState` con una función handler requiere `e.preventDefault()`, manejo manual del loading state, y no funciona sin JS. `useActionState` es la opción correcta para formularios con Server Actions; `useState` es para estado local que no involucra server mutations. `nextjs.org/docs/app/building-your-application/data-fetching/server-actions-and-mutations`

### ¿Qué es Turbopack y en qué estado está en Next.js 15?
Turbopack es el nuevo bundler de Next.js escrito en Rust, diseñado como reemplazo de Webpack. En Next.js 15, está estable para desarrollo (`next dev --turbopack`) con mejoras de velocidad de 45-76% en Fast Refresh y 57% en startup en proyectos grandes. Para producción (`next build`), Turbopack sigue siendo experimental. La API de configuración tiene diferencias con Webpack: no todas las Webpack plugins son compatibles. Para la mayoría de proyectos nuevos, activarlo en dev es seguro y el beneficio en DX es inmediato. `nextjs.org/docs/app/api-reference/turbopack`

### ¿Cómo manejarías metadata dinámica (Open Graph, title) en App Router?
Exportando una función `generateMetadata` async desde `page.tsx` o `layout.tsx`. Esta función recibe los `params` de la ruta y puede hacer fetch para obtener el título real, descripción e imagen OG de un recurso dinámico. Next.js deduplica el fetch con Request Memoization si el mismo fetch ya se hizo en el componente de la página. El fallback es exportar un objeto `metadata` estático. En entrevista, el punto a enfatizar es que `generateMetadata` corre en el servidor y puede acceder a la DB directamente, sin necesidad de un endpoint separado. `nextjs.org/docs/app/building-your-application/optimizing/metadata`

### ¿Por qué no deberías poner lógica de negocio sensible en un Client Component?
Porque todo el código de un Client Component se incluye en el bundle de JS que el browser descarga. Variables de entorno sin `NEXT_PUBLIC_`, conexiones a DB, claves privadas, cualquier cosa en un Client Component es visible en el source code del bundle. La lógica sensible va en Server Components, Server Actions o Route Handlers que se ejecutan exclusivamente en el servidor. Esta es una diferencia crítica respecto al Pages Router donde era fácil confundirse: en el App Router el límite servidor/cliente es explícito en el código. `nextjs.org/docs/app/building-your-application/rendering/client-components`

### ¿Cómo implementarías optimistic updates con Server Actions?
Con el hook `useOptimistic` de React (estable en React 19 / Next.js 15). `useOptimistic` toma el estado actual y una función reductora, y expone una versión optimista del estado que se actualiza inmediatamente al llamar `addOptimistic`. Cuando la Server Action completa (con éxito o error), el estado optimista se reconcilia con el estado real. Si la action falla, el estado vuelve al valor original automáticamente. Es el reemplazo del patrón manual de setState + try/catch + rollback. `nextjs.org/docs/app/building-your-application/data-fetching/server-actions-and-mutations#optimistic-updates`

### ¿Qué configuración de `next/image` causa más impacto negativo en LCP?
No agregar `priority` a la imagen LCP (above the fold) es el error más común. Sin `priority`, la imagen carga de forma lazy: el browser la descubre tarde en el ciclo de rendering y la carga cuando ya está visible, lo que dispara el LCP tarde. El segundo error es no configurar `sizes` correctamente: sin `sizes`, el browser asume el peor caso (imagen a 100vw) y puede descargar una imagen mucho más grande de la necesaria. El tercer error es usar `layout="fill"` sin un contenedor de altura definida, causando CLS. `nextjs.org/docs/app/building-your-application/optimizing/images`
